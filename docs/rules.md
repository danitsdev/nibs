# Nibble Rules Specification

Nibble uses community-extensible YAML rule files to define how junk, caches, and build artifacts are detected. Rules reside in the `rules/` directory and are loaded or compiled into the Nibble binary.

## Rule Schema

A typical rule contains the following fields:

```yaml
id: node_modules
name: Node dependencies
category: rebuildable_dependency
risk: safe
patterns:
  - "**/node_modules"
reason: "Node dependency folders are generated by package managers and can usually be restored."
restore:
  - "npm install"
  - "pnpm install"
  - "yarn install"
default_action: review
```

### Field Definitions

- **`id`** (string): Unique identifier for the rule.
- **`name`** (string): Human-readable title of the rule.
- **`category`** (string): Category classification. One of:
  - `rebuildable_dependency`
  - `build_artifact`
  - `framework_cache`
  - `language_cache`
  - `temporary_file`
  - `trash`
  - `exact_duplicate`
  - `large_unknown`
  - `docker_info`
  - `system_info_only`
- **`risk`** (string): Risk classification. One of: `safe`, `review`, `risky`, `info`.
- **`patterns`** (array of strings): Glob-like patterns or directory names matching targets.
- **`reason`** (string): Detailed explanation of what the folder/file is and why it can be cleaned.
- **`restore`** (array of strings, optional): Terminal commands the developer can run to rebuild or restore the deleted directory.
- **`default_action`** (string, optional): Default action. One of: `clean`, `review`, `keep`.

---

## Writing Custom Rules

To add a new rule:
1. Create or edit a YAML file in the `rules/` directory (e.g. `rules/custom-tools.yaml`).
2. Implement the schema shown above.
3. Keep patterns focused. Instead of matching broad directories (like `**/.cache`), target specific tool subdirectories (e.g. `**/.cache/my-tool-cache`).

## Default Rules Pack

Nibble comes pre-configured with rules for common developer suites:

- **Node/Bundlers** (`rules/node.yaml`): Node modules, Next.js/Nuxt.js build caches, Vite caches, Turborepo, and Parcel caches.
- **Python** (`rules/python.yaml`): Bytecode caches, pytest, mypy, ruff linting caches, Virtual Environments (`.venv`), and tox test setups.
- **Rust** (`rules/rust.yaml`): Project build targets (`target/`), Cargo global registry caches, and local git database caches.
- **Go** (`rules/go.yaml`): Go build cache directories.
- **Java/JVM** (`rules/java.yaml`): Gradle build caches, Maven package dependency repository caches.
- **Developer Extras** (`rules/developer-extras.yaml`): Zig compile caches, Dart/Flutter caches, Angular builder caches, SvelteKit environments, Astro build caches, Swift Package Manager local builds, React Native Android caches, Expo metadata caching, Bun install folders, Yarn local package downloads, and global PNPM stores.
- **Application Caches** (`rules/application-caches.yaml`): Large non-developer consumer application caches including Google Chrome, Mozilla Firefox, Spotify music caching, Discord chat assets, Slack team workspaces, and Steam pre-compiled Vulkan/OpenGL shader caches.
- **Linux & Ecosystem** (`rules/linux-cache.yaml`): FreeDesktop Trash, system temp files (`/tmp`, `/var/tmp`), **Codex Update Manager workspaces, downloads and runtimes**, **Playwright Go browser binaries**, system package manager caches (APT, DNF, Pacman), systemd journal archives, rotated log files (`*.gz`, `*.[0-9]`), Flatpak user application caches, and Snap package revision downloads.
